# Validation Strategy

## 1. Principle

Validation exists at every trust boundary. Frontend checks improve usability, while backend checks are
authoritative because a request can bypass or modify the browser application.

The backend uses Zod for environment values, JWT payload shape, authentication and analysis bodies,
history queries, and submission identifiers. The frontend uses focused TypeScript validation helpers
for the implemented forms and editor. The two layers intentionally mirror constraints without sharing
runtime code across the deployment boundary.

## 2. Validation Locations

### 2.1 Frontend

Frontend helpers validate registration and login fields before submission. The execution hook rejects
blank source and source over 100,000 characters. Pagination values are generated by the history hook
rather than accepted as arbitrary user text. Accessible field messages and focus management help the
student correct invalid forms.

Frontend checks never hash passwords, verify JWT signatures, determine ownership, classify Python
errors, or replace backend responses.

### 2.2 Backend HTTP boundary

- Registration and login bodies use strict Zod schemas in `auth.validator.ts`.
- Analysis bodies use a strict Zod schema in `analysis.validator.ts`.
- History query values and submission identifiers are parsed by Zod in `history.validator.ts`.
- Bearer headers are parsed by authentication middleware, and verified JWT payloads are checked by
  Zod before database identity resolution.

Invalid requests stop before password hashing, database writes, Docker execution, or AI calls. Zod
issues become the standard `400 VALIDATION_ERROR` envelope; rejected values and raw Zod objects are not
returned or logged.

### 2.3 Environment boundary

The backend validates its complete environment once during startup. Missing MongoDB/JWT values,
unsupported provider names, unsafe limits, a missing Gemini key, or a sample production JWT secret
prevent startup with a safe field-name summary.

The frontend's `BACKEND_API_URL` is server-only Next.js configuration. Browser requests remain
same-origin and no backend origin or secret is exposed through a `NEXT_PUBLIC_` variable.

### 2.4 Internal adapter boundaries

The Python runner is an in-process typed backend module, not an external response parser. It constructs
a complete `RunnerResult` through controlled classification functions. Docker identifiers and parsed
container state receive runtime checks before use. AI provider output is trimmed, required to be
non-empty, and limited to 20,000 characters before persistence.

### 2.5 Database boundary

Mongoose schemas enforce required fields, enums, string lengths, normalized email behavior, types, and
timestamps. The unique email index is the final duplicate-account safeguard. Database validation
complements the HTTP schemas but is not the first line of request validation.

## 3. Canonical Rules

| Value                   | Rules                                                                              | Transformation                            |
| ----------------------- | ---------------------------------------------------------------------------------- | ----------------------------------------- |
| Registration `name`     | Required string, 2–100 characters after trimming                                   | Trim before storage                       |
| `email`                 | Required valid email, maximum 254 characters                                       | Trim and lowercase                        |
| Registration `password` | Required string, 8–64 characters, maximum 72 UTF-8 bytes                           | Preserve exact value, then bcrypt hash    |
| Login `password`        | Required string, 8–64 characters, maximum 72 UTF-8 bytes                           | Preserve exact value, then bcrypt compare |
| Analysis `code`         | Required string, at least one non-whitespace character, maximum 100,000 characters | Preserve exact source                     |
| History `page`          | Positive integer; default `1`                                                      | Zod numeric coercion                      |
| History `limit`         | Integer 1–50; default `10`                                                         | Zod numeric coercion                      |
| Submission `id`         | Exactly 24 hexadecimal characters                                                  | Convert only after validation             |

Passwords and source code are never trimmed. The nonblank code check inspects a trimmed copy while
preserving the program's original indentation and characters.

## 4. Strictness and Error Mapping

Request bodies and history queries are strict. Unknown properties return `400 VALIDATION_ERROR`
instead of being silently accepted. Every Zod issue maps to an item with:

- `code: "VALIDATION_ERROR"`;
- an optional dot-notation `field` path;
- a safe rule-specific `message`.

Malformed JSON uses `INVALID_JSON`, an oversized body uses `PAYLOAD_TOO_LARGE`, and an invalid history
identifier uses `INVALID_SUBMISSION_ID`.

## 5. Synchronization Rule

A changed input constraint requires coordinated updates to backend schemas, frontend helpers, API and
validation documentation, tests, examples, and user-facing messages. The backend remains authoritative
if the layers ever disagree.
